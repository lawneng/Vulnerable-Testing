title: Security Review – Insecure SQL Construction in `doPost`
---

## Summary
The servlet builds an SQL statement by directly concatenating a value that originates from the HTTP request header **`BenchmarkTest00200`**. The value is URL‑decoded, placed into a `List`, and ultimately assigned to the variable `bar`. The final query is:
```java
String sql = "SELECT * from USERS where USERNAME='foo' and PASSWORD='" + bar + "'";
```
No input validation, escaping, or use of prepared statements is performed. This is a classic **SQL Injection** vulnerability.

## Impact
| Aspect | Description |
|---|---|
| **Confidentiality** | An attacker can retrieve other users’ data, or even dump the entire `USERS` table by injecting `OR 1=1` or similar conditions. |
| **Integrity** | By terminating the original statement with a quote and appending additional statements (e.g., `; UPDATE USERS SET PASSWORD='hacked' WHERE USERNAME='admin'`), an attacker can modify or delete data. |
| **Availability** | Supplying a very long header value or malformed UTF‑8 can cause the servlet to throw exceptions, potentially leading to a denial‑of‑service. |
| **Privilege Escalation** | If the database account used by `DatabaseHelper.JDBCtemplate` has elevated rights, the attacker may gain full control over the database. |

Using the Spring `batchUpdate` method does **not** mitigate the risk – it simply executes the supplied string as a batch of update statements.

## Exploitation Scenarios
1. **Simple Bypass of Authentication**
   ```http
   BenchmarkTest00200: ' OR '1'='1
   ```
   The resulting SQL becomes:
   ```sql
   SELECT * from USERS where USERNAME='foo' and PASSWORD='' OR '1'='1'
   ```
   `WHERE` clause always true → the query returns all rows, effectively bypassing authentication.
2. **Data Exfiltration**
   ```http
   BenchmarkTest00200: ' UNION SELECT credit_card, ssn, null, null FROM CREDIT_CARDS --
   ```
   If the database engine permits `UNION` in a `SELECT` constructed this way, the attacker can pull sensitive data from other tables.
3. **Command Injection** (multi‑statement)
   ```http
   BenchmarkTest00200: '; DROP TABLE USERS; --
   ```
   The final SQL becomes:
   ```sql
   SELECT * from USERS where USERNAME='foo' and PASSWORD=''; DROP TABLE USERS; --'
   ```
   Executed as two statements, deleting the `USERS` table.

## Additional Issues
- **Header‑based Input**: Using an HTTP **header** for authentication data is unconventional and may be logged by intermediate proxies, increasing exposure.
- **URL Decoding without Validation**: `java.net.URLDecoder.decode` will accept any percent‑encoded sequence, potentially leading to malformed Unicode that could trigger exceptions.
- **Error Handling**: When `hideSQLErrors` is false the full stack trace is re‑thrown as a `ServletException`, leaking internal details.
- **XSS Mitigation**: The SQL string is HTML‑encoded before being echoed, which is good, but the real problem is the query itself.

## Recommendations
1. **Use Prepared Statements** (or Spring’s `JdbcTemplate` with `?` placeholders):
   ```java
   String sql = "SELECT * FROM USERS WHERE USERNAME = ? AND PASSWORD = ?";
   jdbcTemplate.query(sql, new Object[]{"foo", bar}, ...);
   ```
2. **Validate and Sanitize Input** – reject any characters outside an expected whitelist (e.g., alphanumerics) before using it.
3. **Never Build Queries from HTTP Headers** – move authentication data to the request body or standard form parameters.
4. **Limit Database Privileges** – the account used for this query should have *read‑only* rights for the `USERS` table, preventing data modification.
5. **Centralise Error Handling** – always hide database errors from the client and log them securely on the server.
6. **Log Input Carefully** – avoid logging raw header values that may contain injection payloads.
