title: Security Review of `svc_rpc_gss_validate`
---

## Vulnerability
The function validates the credential length with:
```c
if (oa->oa_length > sizeof(rpchdr) - 8 * BYTES_PER_XDR_UNIT) {
    …
    return FALSE;
}
```
`oa_length` is stored as a **signed 32‑bit integer** in the `opaque_auth` structure. The check only rejects *large positive* values; **negative values pass** the test because `-1 > 96` evaluates to false. The subsequent call to
```c
memcpy((caddr_t)buf, oa->oa_base, oa->oa_length);
```
converts the signed `oa_length` to `size_t`. A negative value becomes a huge unsigned size (e.g. `0xffffffff` → 4 GB), causing `memcpy` to copy far beyond the 128‑byte `rpchdr` buffer on the kernel stack.

### Consequences
- **Stack buffer overflow** – the overflow can overwrite saved registers, the saved frame pointer, and the return address.
- The overflow occurs in a kernel function executed in **ring 0** by the NFS worker thread.
- No stack canary is present (the buffer is `int32_t`, not `char[]`), and KASLR is disabled on FreeBSD 14.x, so the return address is at a known offset.

## Severity Assessment
- **Impact:** Remote code execution with full kernel (root) privileges.
- **Exploitability:** High – a single crafted RPCSEC_GSS request triggers the overflow.
- **CVSS v3.1 (approx.):** 9.8 (Critical).

## Exploitation Sketch
1. **Craft a malicious RPC request** where `oa_length` is set to **‑1** (0xffffffff) and `oa_base` points to attacker‑controlled data (e.g., a buffer in the packet).
2. The server runs `svc_rpc_gss_validate`; the length check passes because the value is negative.
3. `memcpy` copies ~4 GB of data onto the stack, overwriting the saved return address.
4. The attacker places a **kernel‑mode ROP chain** in the overflow payload. Because the kernel base address is fixed (no KASLR) and the stack is writable, the attacker can compute gadget addresses statically.
5. Overwritten RIP points to the first gadget, which pivots to the rest of the chain. Typical chain:
   - `pop rdi ; ret` → `0` (setuid(0))
   - Call `setuid`
   - `pop rdi ; ret` → address of `"/bin/sh"` string placed in the payload
   - `pop rsi ; ret` → `0`
   - `pop rdx ; ret` → `0`
   - Call `execve`
6. Execution of the ROP chain spawns a root shell (or performs any privileged action). The thread then returns cleanly to avoid a kernel panic.

## Mitigation Recommendations
1. **Treat `oa_length` as unsigned** and reject any value that is not within `[0, sizeof(rpchdr) - header_size]`.
2. Add an explicit **non‑negative check** before the size comparison.
3. Use a **stack‑canary** that also protects functions with non‑char arrays (e.g., compile with `-fstack-protector-strong`).
4. Consider moving the reconstruction buffer to the heap or using a fixed, larger static buffer that can safely accommodate the maximum allowed credential size (`MAX_AUTH_BYTES`).
