## Vulnerability Summary
The function **`svc_rpc_gss_validate`** copies the credential payload (`oa->oa_base`) directly into a fixed‑size stack buffer (`rpchdr`) without validating the length (`oa->oa_length`). The buffer is only 128 bytes (32 uint32_t entries). An attacker can supply a credential with `oa_length` greater than the remaining space, causing a **stack buffer overflow**.

## Severity Assessment
- **Impact:** Remote code execution (RCE) with the privileges of the NFS server process (typically `root`).  
- **Exploitability:** High – the overflow is triggered by a single RPC request; no authentication is required beyond the normal RPCSEC_GSS handshake, which can be bypassed by supplying a malformed credential.
- **CVSS v3.1 (approx.):** 9.8 (Critical).

## Exploitation Scenario
1. An attacker crafts an NFS client that sends an `RPCSEC_GSS` request containing a credential (`oa`) whose `oa_length` is larger than the remaining space in `rpchdr` (e.g., **200 bytes**).  
2. The server executes `memcpy((caddr_t)buf, oa->oa_base, oa->oa_length);` copying the oversized payload onto the stack, overwriting local variables, saved frame pointer, and the return address.
3. By controlling the overwritten return address, the attacker redirects execution to shellcode or a ROP chain placed in the payload, achieving arbitrary code execution on the NFS server.
4. Because the NFS server runs with elevated privileges, the attacker gains full system compromise.

## Recommended Fix
```c
    /* Enforce a hard limit before copying */
    const int32_t max_len = sizeof(rpchdr) - (int)((char *)buf - (char *)rpchdr);
    if (oa->oa_length > max_len) {
        /* reject the request */
        return FALSE;
    }
    if (oa->oa_length) {
        memcpy((caddr_t)buf, oa->oa_base, oa->oa_length);
        buf += RNDUP(oa->oa_length) / sizeof(int32_t);
    }
```
*Alternatively*, allocate the header dynamically based on `oa->oa_length` or use a safe copy function that respects the buffer size.
